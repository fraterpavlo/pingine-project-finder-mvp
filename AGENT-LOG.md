# AGENT-LOG — общий журнал агентов ProjectFinder

> Это не отчёт. Это **рабочая память** между сессиями агентов.
> Перед началом работы — прочитай последние 30–50 строк.
> После завершения работы — добавь короткую запись (формат ниже).
>
> Содержательные правила и контракты — в [CLAUDE.md](CLAUDE.md).
> Полная история ревью с ID проблем — в [CONSOLIDATED-REVIEW.md](CONSOLIDATED-REVIEW.md).
> Здесь — только текущий контекст и грабли.

---

## Формат записи

> **Полный список «что писать / что не писать» — в [CLAUDE.md](CLAUDE.md), секция «Принципы записи в AGENT-LOG.md».**
> Сюда попадает только то, что пригодится будущему агенту: архитектурные решения, грабли, контракты, отвергнутые альтернативы, открытые вопросы. Рутинные операции, артефакты работы, перечисление файлов — НЕ сюда.

Новые записи — наверху секции «Журнал». Шаблон, 3–8 строк максимум:

```
### YYYY-MM-DD — короткое название (одна строка)
**Решение / находка:** что важного зафиксировали (1–3 строки).
**Почему:** короткое обоснование (1–2 строки).
**Действие для будущего:** что следующий агент должен учесть (если есть).
```

Если получается больше 8 строк — это отчёт оператору, а не заметка для агента. Перенеси в CONSOLIDATED-REVIEW.md, оставь здесь только ссылку и одну строку «суть».

Перед сохранением задай три вопроса (см. CLAUDE.md). Если на все три «нет» — не записывай.

## Известные грабли (читать ВСЕГДА перед работой)

> Это не список «что плохо», а **минное поле** для будущих агентов. Каждый пункт — реальная история, которую нашли через дебаг или ревью.

1. **`projectfinder.sqlite` через FUSE = смерть БД.** Cowork-скилл НИКОГДА не открывает этот файл. Ни на чтение, ни на запись. Только `data/snapshot.sqlite` (read-only) + `pf_intents.emit(...)` (запись через очередь). История — [CONSOLIDATED-REVIEW.md §2](CONSOLIDATED-REVIEW.md). Любой `import pf_db` в Cowork-скилле — красный флаг.

2. **Identity HR-письма.** `From:` в SMTP формирует `email_io.resolve_from_identity` из `developers/<id>.json → fixed.email_identity` (НЕ из `email-config.json`, там только транспорт). Если сделаешь `prof.get("email_identity")` — получишь `None`, потому что поле вложено в `fixed`. Был P0-NEW-1, чинился одной строкой.

3. **`evaluate-job` возвращает `score_letter`, не `score`.** Это согласовано с `evaluate-and-initiate`. Любая правка одного скилла без второго → молчаливый рассинхрон.

4. **`generate-draft` — чистая функция.** Никаких файлов в `data/drafts/`, никаких Gmail-черновиков через MCP. Эти побочки были в JSON-эре, удалены. Возвращай словарь — и всё.

5. **`dialogue-agent` берёт conversation_id из `incoming.conversation_id`.** Демон (`email_io.append_incoming` / `telegram_io.append_incoming`) уже привязал его к существующей conversation. Поиск через snapshot — только fallback для «сирот»; иначе будет race c read-lag → дубль conversation.

6. **conversation должна быть найдена ДО создания.** В `evaluate-and-initiate` — генерируем `conv-<hex8>`, в `dialogue-agent` — берём из incoming. Никаких `conv-{job_id}-001` шаблонов с разным форматом (был P0-4).

7. **Связку `create_conversation + insert_outgoing + append_conversation_message` эмить ОДНИМ `emit_batch`.** Иначе `ops_applier` может применить только часть, и dialogue-agent получит outgoing без записи в `conversation_messages` → ответит «в пустоту».

8. **`pf_policy.decide_outgoing_status` — единственный источник правды для auto-send.** Не дублируй логику в SKILL.md. Если меняешь правило — меняешь функцию, остальные инлайнят её.

9. **`is_admin: true` в `notifications-config.json`** определяет получателя системных алертов. Не `notify_on=["admin"]`, не угадай. Старые конфиги могут быть без этого поля — `pf_db.notify_admin` имеет fallback (по `notify_on` lower-case + первый chat_id).

10. **`RECOVER_STUCK_AFTER_SEC = 600`** в `email_io.py` и `telegram_io.py`. Не уменьшай ниже 600 — `HUMAN_DELAY_RANGE` в TG может занять до 180с, и узкий буфер = дубли отправки. См. P1-NEW-8.

11. **Advisory-lock в `evaluate-and-initiate`** через `service_state.evaluate_and_initiate.lock` с TTL 30 мин. Параллельные запуски (cron + ручной) могут эмитить одни и те же intents для тех же jobs — HR получит два письма. Лок защищает.

12. **Cycle-summary использует детерминированный `idempotency_key`** (sha1 от 30-минутного окна). Не передавай свой uuid — ops_applier пометит дубль автоматически.

13. **scan-sources в Cowork сейчас ВЫКЛЮЧЕН** (по согласованию с оператором, Q3 от 2026-04-16). В `sources.json` пока только тестовый TG-канал, который и так обрабатывает локальный `telegram_scanner.py`. Включать `scan-sources` обратно — только после первой реальной web-источника.

14. **Только один профиль разработчика сейчас** — `test-fullstack.json` с пометкой `replace_before_production: true`. Multi-developer Telegram (одна сессия per dev) пока не реализован.

15. **Демоны нельзя останавливать «жёстко»** во время ops_applier-цикла — есть риск intent в `pending/` без attempt'а на applied. Но `Ctrl+C` корректно даёт ops_applier завершить текущий файл.

16. **`email_identity.from_address` ДОЛЖЕН совпадать с `email-config.json → smtp.username`.** Иначе Gmail SMTP перепишет `From:` на authenticated-адрес или добавит «via suprrama@gmail.com» — HR увидит «не того» отправителя. Хуже: HR ответит на адрес из `From:`, а IMAP-listener слушает другой ящик → ответ потеряется. Если хочется реально слать с другого адреса — нужно настроить «Send mail as» в Gmail-настройках того аккаунта (отдельная процедура с подтверждением). Сейчас оба поля = `suprrama@gmail.com`. Display name (`from_name`) — произвольный, его Gmail не трогает.

17. **Cascade-чек ОБЯЗАТЕЛЕН** при любом изменении архитектуры/контракта/имени поля. Точечная правка без каскада — главный источник регрессий. Подробный чек-лист — в [CLAUDE.md](CLAUDE.md) секция «Cascade-проверка». Минимум: `Grep` по имени изменённой сущности по всему репо + sync везде, где найдено.

18. **FUSE-mount Cowork-сессии может показывать stale/обрезанные/null-padded версии файлов**, которых на Windows-диске нет. Долгоживущая Cowork-сессия особенно подвержена. Симптомы: `wc -l` в Cowork ≠ реальное число строк на Windows; `xxd | tail` показывает NUL-хвост; `diff <(git show HEAD:X) X` выдаёт мусор. **Правило безопасности:** если `diff <(git show HEAD:X) X` непустой — НЕ пиши в `X` через FUSE (затрёт актуальный Windows-файл stale-версией). Читай через `git show HEAD:<path>` (git-object store обходит FUSE-кеш). Если правки накопились на Windows — закоммить их перед тем, как давать Cowork-агенту команду на чтение.

## Открытые вопросы

> То, что НЕ закрыто и требует решения. Если решил — перенеси в журнал и удали отсюда.

- **Реальные dev-профили.** Когда выходим в бой — заменить `test-fullstack` на реальный профиль. Структура файла — см. сам `test-fullstack.json` (он самодокументированный). См. P2-8 в CONSOLIDATED-REVIEW.
- **Heartbeat для всех демонов.** Сейчас только `ops_applier` пишет heartbeat. Нужно добавить такой же `state_set("heartbeat.<service>", utcnow_iso())` в остальные пять. Команда `/status` в боте сравнит с `now - 5min` и алертнёт. См. P2-3.
- **Dry-run режим.** Env-var `PF_DRY_RUN=1`. SMTP/Telethon send пропускается, в лог `[DRY RUN] would send to ...`. Полезно перед первым реальным HR. См. P2-4.
- **Автобэкапы БД.** Раз в сутки в `ops_applier` — `VACUUM INTO 'data/backups/YYYY-MM-DD.sqlite'`, ротация 7 последних. Папка уже в `.gitignore`. См. P2-5.
- **JSON-логи с trace_id.** Каждый intent → trace_id, каждое его применение → лог с тем же trace_id, каждое исходящее → лог. При проблеме `grep trace_id` показывает всю цепочку. См. P3.
- **Шифрование `conversation_messages.content`.** Там реальные ответы HR. Fernet-ключ из `secrets.json`. Не блокер для тестового прогона, но нужно до первого реального продакшен-HR.
- **Multi-developer Telegram.** Сейчас single-account. Когда появится второй разработчик — `TelegramClient` per developer + `session_<dev_id>` файлы + диспатч в `process_outgoing` по `outgoing.developer_id`. См. P2-NEW-2.
- **Динамическая адаптация резюме под вакансию.** В планах после стабилизации базовой работы.

## Журнал

### 2026-04-16 — `state_set(value=None)` теперь делает DELETE; ops_applier строже к IntegrityError
**Решение / находка:** (а) `pf_db.state_set` при `value=None` теперь делает `DELETE FROM service_state WHERE key=?` вместо `INSERT NULL` (схема имеет NOT NULL). Это семантически правильно: «снять» = «забыть». (б) `ops_applier._apply_intent` глотал ВСЕ `IntegrityError` как «UNIQUE-дубль» → реальные NOT NULL/FK/CHECK ошибки молча терялись. Теперь идемпотентным считается только `IntegrityError` со словом `unique` в тексте; остальные → `failed`, intent уходит в `intents/failed/`.
**Почему важно:** advisory-lock в evaluate-and-initiate снимается через `state_set("lock", None)`. Со старым кодом это давало `NOT NULL constraint failed`, ops_applier помечал applied — лок мог застрять. Будущие операции с value=None (любые «снять/забыть» через intent) теперь работают без обхода.
**Действие для будущего:** если пишешь новую state-операцию — value=None семантически безопасно. При диагностике — смотри `intents/failed/`, теперь там всплывут реальные нарушения схемы, а не только парс-ошибки.

### 2026-04-16 — evaluate-and-initiate: ранний выход из-за FUSE-артефактов
**Решение / находка:** `snapshot.sqlite` малформирован (читается через FUSE как испорченный файл), `notifications-config.json` имеет NUL-паддинг в хвосте (640 нул-байт) — `json.loads` падает с «Extra data». Оба файла — жертвы грабли #18. ops_applier при этом запущен и нормально применял intents за ~5с.
**Почему важно:** grabля #18 в документации упоминает только snapshot и «любые файлы», но конкретный пример с NUL-padded config впервые зафиксирован здесь. При парсинге конфигов из Cowork — добавляй `raw.rstrip('\x00')` перед `json.loads`.
**Действие для будущего:** при «Extra data» в JSON-конфиге — проверь NUL-хвост: `repr(raw[-100:])`. Если да — `json.loads(raw.split('\x00')[0])`.

### 2026-04-16 — выбор архитектуры доступа к БД из Cowork (Variant A vs B)
**Решение:** intent-queue + read-only snapshot (Variant A). Отвергнут journal_mode=DELETE + прямой FUSE-доступ (Variant B).
**Почему:** copy-back .sqlite через FUSE необратимо рассинхронизирует .sqlite и .sqlite-wal → битые индексы, потеря данных демонов. Подробный разбор обоих вариантов и сравнительная таблица — в [CONSOLIDATED-REVIEW.md §2.4–2.6](CONSOLIDATED-REVIEW.md). Variant B оставлен только как «прямо сейчас залатать на 2 часа».
**Действие для будущего:** ЛЮБОЙ Cowork-скилл с `import pf_db` или прямым `sqlite3.connect(projectfinder.sqlite)` — баг. Только snapshot (read) + pf_intents (write). См. граблю №1.

### 2026-04-16 — отвергнутые альтернативы и согласованные правила (Q1–Q7)
**Решения:** Q3 — `projectfinder-scan-sources` отключён в Cowork до появления реальных web-источников (sources.json содержит только тестовый TG-канал, его обрабатывает локальный telegram_scanner.py). Q5 — переводы скиллов делает один чат, логику правит другой; не пересекаются. Q6 — `php-backend` убран из всех профилей, его в positions.json нет.
**Действие для будущего:** не включай scan-sources в Cowork до явного действия оператора (грабля №13).

### 2026-04-15 — миграция JSON-файлы → SQLite (commit c70fb0e)
**Решение:** всё runtime-состояние перенесено в SQLite (9 таблиц, WAL, two-phase commit, UNIQUE-констрейнты).
**Почему:** конкурентная запись JSON-массивов из 6 процессов + Cowork ломала файлы (incoming.json уже восстанавливался из corruption). JSON-эра была фундаментально нестабильна.
**Действие для будущего:** не возвращайся к JSON для runtime-данных. Если видишь legacy-упоминание `*.json` в `data/` — это, скорее всего, артефакт старой эры, проверь по schema в pf_db.py.
