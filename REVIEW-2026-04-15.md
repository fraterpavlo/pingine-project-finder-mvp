# Ревью ProjectFinder — 2026-04-15

## TL;DR

Система задумана верно, но **не готова к регулярной боевой работе**. Главная проблема уже материализовалась — повреждение JSON-файлов при параллельной записи (`incoming.json` уже восстанавливался из corruption). Причина — архитектурная: несколько процессов (локальные демоны + cloud scheduled task) пишут в одни и те же JSON-массивы, и механизм блокировок в `pf_common.py` их не синхронизирует между собой.

Ниже — конкретные проблемы с приоритетами, оценка надёжности подхода и чек-лист исправлений.

---

## 1. Как устроена система (кратко)

Два слоя:

**Cloud-слой (Cowork scheduled task)** — раз в минуту вызывает скилл `dialogue-agent`, читает `incoming.json`, генерирует ответы, пишет в `outgoing.json` и `conversations.json`.

**Локальные демоны (Python, управляются `projectfinder.py`)**:

- `telegram_scanner.py` — сканирует TG-каналы (telethon), пишет в `found-jobs.json`.
- `telegram_io.py` — listener на DM + отправка исходящих с human-like задержкой.
- `email_io.py` — IMAP poll + SMTP send.
- `telegram_notifier.py` — шлёт нотификации в бот.
- `bot_handler.py` — обрабатывает callback-кнопки approve/edit/reject.

**Данные**: JSON-файлы в `project-finder/data/` (`incoming.json`, `outgoing.json`, `conversations.json`, `notifications.json`, `found-jobs.json`, `escalations.json`).

**Статусы**:
- outgoing: `ready` → `sent` / `needs_review` / `failed` / `rejected`
- incoming: `new` → `processed`
- notifications: `pending` → `sent` → `acknowledged`

## 2. Критические проблемы (P0 — чинить немедленно)

### P0-1. Concurrent writes ломают JSON — это главное

Шесть независимых писателей бьются за одни и те же файлы:
1. `telegram_scanner.py` → `found-jobs.json`
2. `email_io.py` → `incoming.json`, `outgoing.json`, `conversations.json`
3. `telegram_io.py` → `outgoing.json`, `incoming.json`, `conversations.json`
4. `telegram_notifier.py` → `notifications.json`
5. `bot_handler.py` → `outgoing.json`, `notifications.json`
6. **Cowork scheduled task** → `incoming.json`, `outgoing.json`, `conversations.json`, `notifications.json`

Блокировки в `pf_common.py` (`save_json_atomic` + lock-файл) работают **только для локальных демонов**. Cowork scheduled task запускается в другой сессии и использует встроенные Read/Write — он **не видит** lock-файлов. Именно поэтому уже случилось повреждение `incoming.json` (в нём остался комментарий «Восстановлено после corruption»), а HANDOFF.md фиксирует это же: «Файлы обрываются посреди строки».

Дополнительные риски даже внутри локальных демонов:
- Lock-таймаут 10 сек недостаточен для IMAP-операций, которые могут висеть 20–30 сек.
- Если процесс завис/умер с лок-файлом — остальные ждут, порядок не гарантирован.
- `os.replace()` на tmpfile под Windows в редких случаях даёт `PermissionError`, если файл открыт на чтение другим процессом.

**Решение — рекомендую на выбор:**

**A. Минимальные правки (быстро, 1–2 дня)** — перейти на **папки-очереди**:
- `data/outgoing/new/<msg_id>.json`, `data/outgoing/sent/<msg_id>.json`, `data/outgoing/failed/<msg_id>.json`
- Смена статуса = `os.rename()` между папками (**всегда атомарно** и **без конкурентов**, потому что каждый файл пишется ровно один раз).
- Аналогично `incoming/new/` → `incoming/processed/`.
- `conversations/<conversation_id>.json` — по одному файлу на диалог, никто кроме одного процесса в момент времени его не трогает.

**B. Правильное решение (2–3 дня) — SQLite**:
- Одна база `data/projectfinder.sqlite`, WAL-режим (`PRAGMA journal_mode=WAL`).
- ACID-транзакции, блокировки уровня СУБД работают одинаково для любых процессов и из Cowork, и локально.
- Решает сразу: гонки, дедупликацию (UNIQUE по `message_id`/`imap_message_id`/`job_url`), версионность, масштаб.

Я бы голосовал за SQLite — JSON-файлы в проекте такой сложности заведомо хрупкие.

### P0-2. Секреты в открытом виде

В `config/email-config.json` лежит app-password Gmail (`"password": "kunf mxfz vrgb iwof"`), в `config/telegram-client-config.json` — api_id/api_hash, в `config/notifications-config.json` — bot_token. Те же значения продублированы в `HANDOFF.md`. Если эта папка когда-нибудь окажется в публичном git — аккаунт скомпрометирован мгновенно.

**Решение:**

- В JSON-конфигах хранить плейсхолдеры: `"password": "${GMAIL_APP_PASSWORD}"`.

## 3. Высокий приоритет (P1)

### P1-1. Потеря/дубликат отправки при краше процесса

Сценарий в `telegram_io.py` (и зеркально в `email_io.py`):
```
1. читаем outgoing, видим status=ready
2. await client.send_message(...)       ← успешно
3. msg["status"] = "sent"                ← процесс упал тут
4. save_json_atomic(outgoing, ...)       ← не выполнилось
```
После рестарта сообщение снова `ready` → уйдёт **второй раз**. Адресат получит дубль.

**Решение — двухфазный коммит:**
1. Взять сообщение, сразу записать `status=sending`, `attempt_started_at=<ts>`, `attempt_id=<uuid>`.
2. Отправить.
3. После успеха — `status=sent`.
4. При старте демона: все сообщения в `sending` старше N минут — это **незавершённая отправка**. Решение: проверить провайдером (IMAP: искать в Sent по Message-ID; TG: запросить историю) и только потом решать — retry или пометить sent.

Плюс **дедупликация на стороне отправки**: хеш `(recipient, body, date)` — если такое ушло за последние N минут, не слать второй раз.

### P1-2. Failed-сообщения не ретраятся

`telegram_io.py` ставит `status="failed"` при ошибке, и на этом всё — никто больше к нему не возвращается. Сообщение молча теряется. То же в `email_io.py`.

**Решение:** поля `retry_count`, `next_retry_at`, exponential backoff (5m → 15m → 60m), после 3 попыток — эскалировать в `notifications.json` с кнопкой «повторить/отменить».

### P1-3. Нет дедупликации входящих писем

`email_io.append_incoming()` не проверяет, нет ли уже записи с таким `imap_message_id`. Если IMAP не пометил письмо как seen (разрыв соединения в середине `STORE`), на следующем скане оно добавится повторно — dialogue-agent ответит дважды.

**Решение:** `SELECT 1 FROM incoming WHERE imap_message_id = ?` (или `any(m["imap_message_id"] == x for m in messages)`), пометка seen **после** записи в inbox, а не до.

### P1-4. incoming → processed не идемпотентен

Если `dialogue-agent` упал после создания outgoing, но до пометки `incoming.status = processed` — следующий запуск обработает сообщение снова и создаст второй outgoing.

**Решение:** порядок должен быть обратный — **сначала** `incoming.status=processing` с `claim_id`, **потом** генерация ответа, **потом** атомарно `incoming=processed` + создание outgoing в одной транзакции (в SQLite это естественно; в файлах — единой записью через rename папок).

### P1-5. `load_json` без try/except

Если файл битый, `json.load` кидает исключение и крашит весь процесс. Нужен graceful fallback: лог + восстановление из последнего `.bak` (см. P2-1) + запись инцидента в эскалации.

### P1-6. Rate limit Anthropic в scheduled task

HANDOFF упоминает, что по квоте задача периодически падает. При падении `incoming` остаётся в `new`, следующий запуск через минуту снова упирается в лимит. Надо:
- Ловить "rate limit" отдельно, откладывать запуск (exponential backoff).
- Логировать в `notifications.json` как предупреждение, а не как ошибку.

## 4. Средний приоритет (P2)

- **P2-1. Нет бэкапов.** Перед каждой записью сохранять `file.json.bak` (последние 3 версии ротируются). Для SQLite — `VACUUM INTO` раз в сутки.
- **P2-2. Структурированные логи.** Сейчас `projectfinder.log` — текст с цветами. Нужен JSON-формат с полями `service`, `level`, `conversation_id`, `message_id`, `trace_id` — иначе отладка distributed-системы невозможна.
- **P2-3. Heartbeat процессов.** Каждый демон раз в 30 сек пишет `data/heartbeats/<service>.json` с timestamp. Внешний watcher (или сам `projectfinder.py`) алертит, если heartbeat старше 3 минут.
- **P2-4. Dry-run режим.** `PROJECTFINDER_DRY_RUN=1` — не отправляет в SMTP/Telegram, только логирует. Обязательно для тестирования изменений в промптах.
- **P2-5. Дубликаты в `found-jobs.json`.** Дедупликация сейчас только по URL. TG-каналы часто дублируют друг друга — нужно добавить fuzzy-match по `(title, company, salary_range)`.
- **P2-6. `found-jobs.json` растёт без границ** (уже 575 КБ). Архивировать старше 90 дней в отдельный файл, либо переехать в SQLite, где выборка за период — одна индексированная query.
- **P2-7. Сиротские черновики.** `data/drafts/*.md` не удаляются, даже когда outgoing уже `sent` или `rejected`. Со временем накопятся сотни файлов. Лучше вообще хранить черновик прямо в outgoing-записи, а не в отдельном файле.
- **P2-8. LinkedIn в `sources.json`** требует открытого Chrome. В scheduled task браузера нет — эта строка будет молча падать. Либо отключить, либо выносить в отдельный «manual» pipeline.
- **P2-9. Timeout на источник.** Если `hh.ru` не отвечает, web_search блокирует весь pipeline. Нужен `timeout_sec` per source.
- **P2-10. Шифрование `conversations.json`.** Там лежат переписки с реальными людьми — это персональные данные. Хотя бы Fernet на весь файл.

## 5. Замечания к конфигам (что добавить / переделать)

### `positions.json`

- Нет `max_usd_monthly` — вакансии с нереалистично высоким числом (часто signal акций / scam) не отсекаются.
- Нет позиций для других стеков (Node.js standalone, Python, Go). `fullstack` это не закрывает.
- `forbidden_in_communications` слишком широкий («любое упоминание компании»). Нужны конкретные стоп-слова и разрешённые обороты.

### `sources.json`

- Три TG-канала с пометкой «тестовые заглушки» — убрать или заменить реальными.
- LinkedIn с `scan_method=browser` — см. P2-8.
- Нет `timeout_sec` и `enabled_in_scheduled` (чтобы помечать, какие источники можно гонять без браузера).

### `auto-reply-config.json`

- Для `LOW confidence` описание словами, но нет явного поля `escalation_channel`, `escalation_file`.
- `rate_limits.max_outgoing_per_hour: 10` — неясно, это наш софт-лимит или API-лимит. Нужен комментарий.

### `developers/*.json` — **самый слабый конфиг**

Именно от него зависит, насколько dialogue-agent звучит живым человеком, а там не хватает:

- **Список конкретных проектов** с длительностью, стеком и 2–3 хайлайтами на каждый. Сейчас есть `work_history` на 3–4 строки — этого недостаточно, когда HR задаёт уточняющий вопрос «расскажи про последний проект».
- **Уровни владения по стеку**: `{"react": "senior", "nextjs": "middle"}`. Без этого агент не знает, как скромно/уверенно писать.
- **Availability**: notice period, таймзона, часы работы, может ли стартовать сразу.
- **Reject-patterns**: что категорически не интересно (gamedev, казино, gambling, крипто-скам, shift work).
- **Легенда и tone of voice** — отдельное поле, длинный текст: биография, почему ищет новый проект, как обычно общается, любимые темы small-talk, какие вопросы задаёт сам, на что реагирует отказом. Сейчас эта часть размазана по `writing-style.md`, но она должна быть **per-developer**, а не общая.
- **Шаблоны типовых ответов** (salary expectations, «пришлите CV», «созвон когда», «почему ушёл с прошлого места»). Лучше один раз зафиксировать — иначе агент каждый раз выдумывает заново и противоречит сам себе в разных диалогах.
- **«Факты, которые нельзя придумывать»** — список привязанных к резюме сертификатов/компаний/дат. Dialogue-agent должен использовать только их, всё остальное — «не помню точных цифр, уточню».

### `scoring-rules.md`

- Нет реальных примеров (A/B/C/Skip с разбором подсчёта).
- Проверка контекста стоп-слов описана словами, но без алгоритма — надо явно прописать: «смотри 5 слов слева от стоп-слова, ищи отрицания».
- Нет правил по часовым поясам и типам контрактов.

### `writing-style.md`

В целом неплохо, но стоит добавить:
- Конкретные anti-patterns с «до/после».
- Правила про эмодзи (где можно, где нельзя).
- Правило «никогда не извиняйся дважды в одном сообщении», «не используй `Best regards` в русских письмах».

## 6. Надёжность статусов — итого

Таблица переходов и рисков:

| файл | переход | риск | что делать |
|---|---|---|---|
| outgoing | `ready → sent` | дубль отправки при краше | двухфазный коммит через `sending` |
| outgoing | `ready → failed` | сообщение умирает | retry с backoff, потом эскалация |
| outgoing | `needs_review → ready` | гонка bot_handler vs scheduled task | атомарная запись per-файл |
| incoming | `new → processed` | обработка дважды | сначала `processing`, потом outgoing, потом `processed` в одной транзакции |
| incoming | новое письмо | дубль записи | UNIQUE по imap_message_id |
| notifications | `pending → sent` | потеря уведомления | тот же двухфазный подход |

Идемпотентность нигде не обеспечена — это и есть корень почти всех рисков.

## 8. Вердикт

- **Архитектура в целом разумная**, разделение cloud (генерация текста) + local (I/O с Telegram/SMTP) оправдано.
- **Конкретная реализация хранения — слабое место номер один**. JSON-массивы с shared access между cloud и local-процессами обречены ломаться, что уже и произошло.
- **Конфиги проработаны на ~60%**: структура правильная, но `developers/*.json` и `scoring-rules.md` надо углублять — иначе от агента будут шаблонные и противоречивые ответы.
- **Безопасность запущена**: секреты в git-папке — самая срочная дыра.

После закрытия P0 и P1 систему можно запускать в регулярный режим с мониторингом. До этого — только в dry-run и с ручным ревью каждого исходящего.
